A genuinely charming marketing surface with a real motion vocabulary — framer-motion entrances with the portal's spring ease, consistent whileHover/whileTap press feedback, blur-in reveals, and proper exits on the modal, splash, promo bar, and route transitions. The aesthetic instincts are good. But it has one real Critical the portal didn't: the CSS prefers-reduced-motion guard only covers CSS animations — every framer-motion animation ignores the user's reduced-motion preference, because there's no MotionConfig or useReducedMotion. Below that, the issues cluster around ubiquity and perpetuity: an infinite gradient shimmer runs on ~14 headings across every page, two CTA styles pulse forever, the survey's custom overlays and step containers snap out with no exit, and one scroll reveal re-fires every time. For a teen-facing landing, the delight is on-brand — the fix isn't less delight, it's making delight a moment instead of an ambient hum, and honoring the accessibility preference for users who opt out.

Issues to Address

🔴 Critical

1 · framer-motion ignores prefers-reduced-motion

The CSS guard at index.css:149 neutralizes CSS animations (splash blobs, the shimmer, the pulse glows) — but framer-motion animates via JS (WAAPI/rAF), which a CSS media query cannot touch. With no MotionConfig or useReducedMotion anywhere, a vestibular-sensitive user who has reduced-motion enabled still gets every staged y translate, every blur-in reveal, and the infinite floating scroll hint. The fix is a one-liner at the app root: wrap the tree in <MotionConfig reducedMotion="user"> — framer-motion then drops transforms/opacity-translate for those users automatically while keeping instant state changes. This is the audit's only Critical.

2 · Survey overlays & step containers snap out (no exit)

The framer-motion components animate out, but the survey's core flow uses tailwindcss-animate animate-in on plain conditional renders — {activeClass && <ClassRatingDialog/>}, {!tutorialDone && <TutorialOverlay/>}, and every step <div className="animate-in fade-in slide-in-from-bottom-2">. animate-in is enter-only: when the dialog closes or you advance a step, the outgoing element is unmounted instantly while the new one slides in — a visible snap on the most-interacted surface in the app. Wrap these in AnimatePresence (or a Radix Dialog, which animates data-state="closed") and give them a subtler exit.

4 · The gradient shimmer runs forever on ~14 headings sitewide

.gradient-text applies gradShift 3s infinite and it's attached to nearly every heading — the hero, all 8 survey titles, the 404, the promo, the schedule, the splash. Even Jhey's golden rule holds here: "the best animation goes unnoticed… and doesn't fatigue on repeat." A perpetual shimmer on every heading stops being delightful and becomes ambient noise that also quietly burns battery. Keep the effect — make it a moment: shimmer once on load (or on the hero only) and settle to a static gradient. That's more premium, not less playful.

🟡 Important

5 · Pulsing CTA glows (.glow-green / .glow-blue)

Both glow classes run an infinite box-shadow pulse (pulse-green/blue 2.8s infinite) and sit on the primary download CTAs, the portal-destination cards, and the sticky bar. A "breathing CTA" is one of the clearest restraint anti-patterns — perpetual motion on the exact elements the user is trying to read and click. The attention pull is real and it never stops. Move the glow to :hover/:focus-visible (or a single one-shot pulse on mount): the element rests until the user engages, then responds.
